# CLAUDE.md ‚Äî World Cup 2026 Predictor & Fantasy Bot

Operational guide for this project. Read this first; it explains the architecture,
how to run/deploy, and the non-obvious gotchas.

## ‚öÝÔ∏è Sync discipline ‚Äî do this EVERY session (mobile, Cowork, local)
This repo is worked on from several places (local, `claude.ai/code` on mobile, Cowork) **and a cloud bot auto-commits `data/*.json` continuously**. To stay in sync:
1. **At the very start of every session, before any work:** `git pull --rebase origin main`. Never edit files (especially `data/db.json`, `data/my_predictions.json`) without pulling first ‚Äî the cloud commits them often.
2. **After committing any change:** `git push` (and `git pull --rebase` first if the push is rejected). Do this as part of finishing the task, not later.
3. **One live session at a time** on the same files ‚Äî concurrent edits in two places cause conflicts.
4. **On a rebase conflict in `data/*.json`:** these are generated/append-only ‚Äî take the incoming version (`git checkout --theirs <file>`) or regenerate, then `git add` + continue. Never hand-merge them.
The user expects sync to "just work" across devices ‚Äî owning the pull-first / push-last flow is part of every task here, not optional.

## What it does
A Hebrew (RTL) system for the 2026 FIFA World Cup that:
1. **Predicts match results** ‚Äî 1X2 + exact score + confidence (Poisson model blended with market odds).
2. **Advises FIFA Fantasy** ‚Äî legal 15-man squad, captain, per-position picks, transfers, relative to the user's real team.
3. **Telegram bot** ‚Äî the user sends a screenshot (lineup or predictions); the bot reads it (Gemini Vision), replies with advice/comparison, then lets the user **chat freely with Gemini** about the lineup (no structured questions).
4. **Learns** ‚Äî ingests real results to refine team strength, and tracks the user's prediction accuracy vs the model over the tournament.

Runs fully in the cloud (GitHub Actions) ‚Äî no PC required.

## Processing the user's KICKOFF prediction screenshots (READ THIS when images are attached)
The user plays a Hebrew football-predictions league in the **KICKOFF** app (mobile-only). They upload screenshots of their predictions; you compare them to the model and track a cumulative "you vs system" score. **When the user attaches KICKOFF screenshots, do this ‚Äî do NOT use Gemini, read them with your own vision** (Gemini Vision is only for the autonomous cloud bot, and it mis-reads RTL/orientation):
1. Each settled card shows: two team flags+names, the user's prediction badge **"◊Ý◊ô◊ó◊©◊™ X-Y"**, the actual score, and a points badge ‚Äî **◊ë◊ï◊ú = +3 (exact)**, **◊õ◊ô◊ï◊ï◊ü = +1 (correct direction)**, **◊§◊°◊§◊ï◊° = 0 (miss)**, **◊ß◊Ý◊° ◊ó◊û◊ï◊® = ‚àí1 (reversed)**. **`‚àí1` applies ONLY when the predicted scoreline is the EXACT mirror of the actual** (predict 2-1, actual 1-2). Any other wrong-direction result is `0` (miss), not ‚àí1 ‚Äî verified with the user (South Africa 1-0 vs his Korea 2-1 scored 0, not ‚àí1). The team shown **first / on the right** (RTL) is the home/first team. **Ignore input/edit screenshots** (editable score boxes, no points badge) ‚Äî only take settled cards.
2. Extract per game: `home`, `away`, `user_home`, `user_away` (in the app's orientation). Names may be Hebrew or English ‚Äî the helper maps them.
3. Run the helper, which computes the model's pick, records to `predictions_log` (`data/my_predictions.json`), settles against `data/db.json` results, and returns the "you vs system" summary:
   ```python
   import kickoff_predictions as kp
   print(kp.process([{"home": "Egypt", "away": "Belgium", "user_home": 0, "user_away": 2}, ...]))
   ```
   (or CLI: `python kickoff_predictions.py '<json-list>'`). Scoring is KICKOFF: exact +3, direction +1, **exact-mirror reversed ‚àí1** (only when the predicted scoreline is the precise reverse, e.g. predict 2-1 ‚Üí actual 1-2), otherwise wrong direction = 0, miss 0 ‚Äî **no independent goals bonus**. `predictor.match_points` / `predictions_log._points` implement this: `‚àí1` requires `ph==aa and pa==ah`.
4. **Cross-check** your extracted `user_home/away` against the card's points badge ‚Äî if `match_points(user, actual)` ‚âÝ the app's badge, you read the orientation backwards; flip it.
5. For already-played games the model is **in-sample** (it learned from those results ‚Üí inflated); the fair comparison is only for predictions saved **before** kickoff. Say so.
6. `git commit` `data/my_predictions.json` when done.

**If the user also uploads the daily reports they received** (Telegram/HTML, with the model's predictions): read them too, extract the **model's predicted score per game**, and pass it as `model_home`/`model_away` in the same `process(...)` call. This is the **fair** comparison ‚Äî the model's pick as it was *before* the game. Without it, `process` recomputes the model, which for already-played games is **in-sample / inflated** (the model has since learned the result), so always prefer the report's prediction when available.

## Architecture (modules)
- `config.py` ‚Äî settings; loads `.env`. Key knobs: `REPORT_UPCOMING_COUNT` (=5 matches in the report), `POSITION_PICKS_PER_POS`, `TRANSFER_CANDIDATES_PER_POS` (=2), `MARKET_BLEND_WEIGHT`, `ODDS_REVEAL_HOURS`.
  - `FANTASY_SOURCES` ‚Äî site/feed names handed to Gemini as grounded-search hints for prices/form/xG (e.g. *Fantasy Football Scout, WhoScored, FBref, Flashscore, Reddit r/FantasyPL, FotMob, #FPL on Twitter/X*). Gemini does not scrape each site ‚Äî it uses them to steer its Google-grounded search.
  - `ODDS_API_KEY` (+ `ODDS_API_SPORT`/`ODDS_API_REGIONS`) ‚Äî **the-odds-api.com is the PRIMARY odds source**: real live bookmaker odds, **one call** for the whole tournament (~30 calls/month, well inside the free 500/month tier), no Gemini-quota dependency. Set the free key in `.env` + GitHub Secrets. If absent ‚Üí falls back to the Gemini consensus, then model-only.
  - `ODDS_SOURCES` ‚Äî the **top-5 sharpest** bookmaker names for the **Gemini-fallback** consensus odds query (Pinnacle, Betfair Exchange, Bet365, Circa, Oddschecker), handed to Gemini as grounded-search hints; the consensus 1X2 blends into the prediction via `MARKET_BLEND_WEIGHT`.
- `utils.py` ‚Äî logging (file is UTF-8), `load_json`/`save_json` (atomic), `_parse_dt`, `safe_get`.
- `scraper.py` ‚Äî data collection. **Primary fantasy source is the official FIFA World Cup Fantasy feed** (`fetch_official_pool` ‚Üí public no-auth JSON at `config.FIFA_FANTASY_PLAYERS_URL`/`_SQUADS_URL`, ~1488 players √ó 48 squads with real price/ownership/form/official points). It's the canonical roster ‚Äî solves nation- and squad-level validity at the source (no Italy, etc.). `official_differentials` derives differentials from it ‚Äî nailed starters under the ownership ceiling (`DIFFERENTIAL_MAX_OWNERSHIP`, raised to 15%), **ranked by scoring chance** (official avg points **multiplied by next-fixture ease** ‚Äî a hard opponent discounts expected output ‚Äî + form, with low ownership as a bonus via `DIFFERENTIAL_WEIGHTS`, not a hard low gate) so picks are high-upside players who happen to be under-owned, not obscure 1%-owned names. **Scouting-bonus aware** (WC Fantasy 2026 rule: a sub-`SCOUTING_BONUS_OWNERSHIP`%-owned player who returns 4+ pts gets +`SCOUTING_BONUS_POINTS`): `_value` adds an expected-scouting-bonus term (`SCOUTING_BONUS_POINTS √ó min(1, eff_points/8)` for sub-5% players) so cheap high-upside differentials rank higher, and each differential carries a `scouting_bonus` bool surfaced as ‚≠ê in the report. **`official_top_picks`** is the complement ‚Äî "stars of the round": **popular premium** players (ownership ‚â• `STAR_MIN_OWNERSHIP`=10%) with an **easy fixture** (`ease ‚â• min_ease`), ranked by points√óease, carrying `opponent`. Stored as `db['top_picks']` (set wherever `differentials` is, in both `collect()` branches + the bot refresh), passed through advice as `top_picks`, and rendered as a "‚≠ê ◊õ◊ï◊õ◊ë◊ô ◊î◊û◊ó◊ñ◊ï◊®" card/section (HTML + Telegram) above the differentials ‚Äî so the report shows both safe stars and high-upside differentials. The availability filter excludes only *known* benched/unavailable players (`injury_status`/`suspension_status`/`expected_start is False`), so it still populates between rounds when no lineup is confirmed (`matchStatus` is null for everyone) ‚Äî availability comes from `status=="playing"`, not lineup confirmation. It takes `fixture_difficulty`, so in `collect()` it runs after results + fixture-difficulty. Gemini grounded-search is now the **fallback/enrichment** layer (xG/xA, odds, results learning). `db['participants']` stores the 48 official squad names; `participating_nations` prefers it so `filter_to_participants` never false-drops a valid team. **Fixtures, results and fixture-difficulty also come from the official feed now** (`fetch_official_rounds` ‚Üí `rounds.json`): `official_matches` (upcoming, carrying the real `stage` GROUP/R32/‚Ä¶), `official_results` (completed, with penalties + stage), `_record_results` (shared EWMA learner used by both official and the Gemini `ingest_results` fallback), `official_fixture_difficulty` (next-opponent strength: 65% **squad quality** from official player prices via `_team_quality` + 35% results-form, so strong squads like Netherlands read as hard even after one noisy round), and `seed_teams_from_squads` (all 48 teams, preserving learned strength, canonicalising to official names). In `collect()` the official feed is primary and **Gemini is bounded to near-window matches** (`_matches_within_days`) for what only it provides ‚Äî injuries/expected-lineups (`fetch_match_context`), consensus odds, and `_enrich_fantasy_data` (xG/xA, penalty-takers). The whole pipeline runs to a complete db with Gemini fully disabled. Remaining Gemini-only: Vision (screenshot reading) + free chat (`telegram_intake`), odds, and xG/xA/injury enrichment. Fallbacks still via **Gemini grounded search**: `GeminiClient` (`ask_json`, `ask_json_image` for Vision). Fetches matches/teams/match-context/fantasy pool. `ingest_results()` learns from finished scores (EWMA into team goals ‚Üí improves **predictions**). `ingest_player_results()` learns from real player performances. It prefers the **official FIFA Fantasy points** (the complex official scoring incl. bonuses, from play.fifa.com) per player; only when that's missing does it fall back to a computed estimate (`_fantasy_points_for`, the basic goals/assists/appearance/clean-sheet formula). The result is EWMA'd into each player's `recent_points` ‚Üí improves **fantasy** picks; auto-adds standout players missing from the pool (`player_results` rows carry an `official` bool). **Both ingest functions now run inside `collect()` (both branches), after the scrape**, so the daily report keeps learning ‚Äî they are not gated to the bot's ~5h refresh. `_enrich_fantasy_data()` pulls prices/form/xG from `FANTASY_SOURCES`. `filter_to_participants(db)` (called in both `collect()` branches) drops players whose nation isn't one of the 48 WC participants ‚Äî `participating_nations(db)` derives the canonical set from `db['teams']` (+ fixtures/results), and `_clean_nation` normalises spelling variants (`_NATION_ALIASES`: e.g. Czech Republic‚ÜíCzechia, Cape Verde‚ÜíCabo Verde) so valid players aren't dropped. This removes e.g. Italy/Ukraine players that Gemini surfaces from general football knowledge. Applies to both `players` and `differentials`; has a safety net (skips if <24 nations known). The pool/differentials prompts also demand official-26-squad membership.
- `fantasy.py` ‚Äî `expected_points` blends **xG/xA** into the goals/assists per-match rates (50/50 via `_attacking_rates` ‚Äî xG is a more stable predictor than raw goals, neutralising finishing luck), then blends each player's `recent_points` (actual recent FIFA-points signal from `ingest_player_results`) at 0.6 model / 0.4 actual¬∑start-prob. **Captain/vice are chosen by `ceiling_points` (upside), not mean EP** ‚Äî a high-variance goal threat is worth more when points are doubled; penalty takers (`penalty_taker` flag) get a goal-rate bump (`config.CAPTAIN_CEILING_WEIGHT`, `PENALTY_TAKER_GOAL_BONUS`).
- `backtest.py` ‚Äî backtesting harness. Replays `db['results']` through `predictor.predict_match` and scores under KICKOFF (`match_points`) vs two naive baselines (always 1-0 home, always 1-1). `run_backtest(db, scoring)` ‚Üí metrics (ppg/exact/direction) per variant; `format_report`. `tune(db, scoring, grid)` sweeps a config grid (default `MAX_XG`, `HOME_ADVANTAGE`, `MARKET_BLEND_WEIGHT`) via a temporary `_override_config` and reports the best ppg **without mutating config**. The backtest scores **`predicted_score`** (the line the report shows, which the market blend influences) ‚Äî so `MARKET_BLEND_WEIGHT` is genuinely tunable; for that, `_record_results` **archives each match's `market_probabilities` into its result row** and the backtest re-attaches them (without archived odds the weight dimension is inert). Use for A/B before deploying a model change. In-sample (model learned from these results) ‚Äî compare *relatively* between variants; tune is overfit-prone, apply only robust+sensible values. CLI: `python backtest.py` / `python backtest.py --tune`. **`maybe_autotune(db)` runs once per day from `main.py` (the daily report, ~08:00 UTC / 11:00 Israel), before predictions** ‚Äî it runs `tune`, and **applies the best config only if it beats current by ‚â•`min_gain` (0.10) ppg** (so it doesn't chase in-sample noise), persisting the chosen knobs to `data/tuning.json`. `config._apply_saved_tuning()` loads that file at import and overrides `MAX_XG`/`HOME_ADVANTAGE` **within sanity bounds** (`config._TUNING_BOUNDS`). Host advantage is a **relative** bonus (`HOST_HOME_BONUS` over `HOME_ADVANTAGE`) so tuning the neutral value never inverts the host logic. **`model_calibration(db)`** scores how well the model's 1X2 probabilities are calibrated: multiclass **Brier** (mean Œ£(p‚àíy)¬≤; 0=perfect, 0.667=uniform 3-class guess) + favorite hit-rate vs avg favorite-confidence (close ‚áí well-calibrated, not over/under-confident). It's in `run_backtest`'s output and printed by `format_report` (live: Brier ‚âà 0.42, favorite 66% hit vs 70% confidence). In-sample diagnostic ‚Äî read it relatively.
- `predictor.py` ‚Äî Poisson model; `predict_all(db)`; blends consensus odds. **Conditional home advantage** via `_home_advantage_for`: host nations (`config.HOST_NATIONS` = USA/Canada/Mexico) get `HOST_HOME_ADVANTAGE` (0.30) when home; everyone else the neutral `HOME_ADVANTAGE` (0.10). Three scorelines per prediction: **`predicted_score`** (what the daily report shows) = a varied/realistic line ‚Äî goal count from **conservatively-rounded** xG (`_round_goals`, half-DOWN: rounds up only when the fraction is >0.5; matches the Poisson mode floor(Œº)) with **`MAX_XG=3.5`** so the count never exaggerates ‚Äî calibrated against 44 real games: a 4.5 cap piled predictions at "4" (17% of teams vs 11% in reality, avg 1.73 vs 1.52); **3.5 brings the average to 1.57‚âàreality and even lifts ppg 1.16‚Üí1.20**. Since `_round_goals(3.6)=4` but `3.5=3`, the ceiling (and `_TUNING_BOUNDS["MAX_XG"]` upper) is held at **3.5** so a team is essentially never predicted above 3, direction from the blended 1X2, and a draw when `draw_prob ‚â• config.DRAW_PREDICT_THRESHOLD` (=0.25, calibrated to the real ~31% draw rate; the EV pick almost never shows draws because draw is rarely the modal outcome). `recommended_score` **maximises expected points** under `config.PREDICTION_SCORING` (the conservative "safe for points" line, shown as a secondary tag). `most_likely_score` = single most-probable score. The model's total-goals average is well-calibrated (‚âà3.0/game = reality); `predicted_score` exists because the EV pick collapsed to 1-0/1-1 and the user wanted more varied, higher-scoring, draw-aware predictions. `_realistic_scoreline` builds it. Each prediction also carries `recommended_ep`, **`total_expected_goals`** (home+away xG) and **`clean_sheet`** `{home, away}` (from `clean_sheet_probabilities(matrix)` ‚Äî home CS = P(away scores 0) = column-0 sum) ‚Äî both surfaced in the daily report (HTML + Telegram) so the report shows goal expectancy + clean-sheet odds per match. When settling user predictions, `predictions_log.settle_with_results` **backfills a missing `date` from the official result row** so dates never need manual patching.
- `lineup_alerts.py` ‚Äî **owned-player lineup alerts**. When a match lineup is published and one of the user's `data/my_team.json` players is **not starting** (`expected_start is False` = benched) or **unavailable** (`injury_status` injured/out, `suspension_status` suspended), it sends a one-off Telegram alert. `check_alerts(pool, squad, already)` dedupes via a `state['lineup_alerts']` name‚Üícode map (re-alerts on status change, resets when the player is back in the XI); `_match` is truncation-tolerant (handles `my_team.json` short/cut names like "Nuno Men..."). `run_lineup_alerts(send_fn, state)` orchestrates (loads squad, pulls the official pool, sends). Wired into `telegram_intake.run_bot_once` **before the intake-disabled gate**, so alerts fire every ~15 min even though bot intake is off. `expected_start is None` (lineup not yet out) ‚Üí no alert.
- `md_fantasy.py` ‚Äî per-matchday full-squad generator CLI (`python md_fantasy.py`). Thin orchestrator over `fantasy.build_fantasy`: `build_matchday_squad(db, predictions)` builds the legal 15-man squad + XI + captain/vice + bench + cost, then annotates each player with `opponent` (from `db['fixture_difficulty']`) and `scouting_bonus` (ownership < 5%); `format_squad` renders it RTL (formation, cost/100M, captain üëë, lines, bench). Reuses the existing engine ‚Äî no duplicate squad logic.
- `odds.py` ‚Äî market-odds aggregation, two sources by priority via **`fetch_market_odds(matches, gemini)`**: (1) **the-odds-api.com** (`fetch_odds_api` ‚Äî real live bookmaker odds, one HTTP call for the whole tournament; events mapped to our matches by normalised team-pair with `_ODDS_ALIASES` for name diffs like South Korea‚ÜíKorea Republic, Ivory Coast‚ÜíC√¥te d'Ivoire), then (2) **Gemini consensus** (`fetch_consensus_odds`) only for matches the API didn't cover. Both go through `remove_vig` + `_is_plausible`; result is `match_id ‚Üí {home_win, draw, away_win, sources}`. Never raises ‚Äî any failure degrades to model-only. `scraper.collect()` calls `fetch_market_odds` at both call sites.
- `fantasy.py` ‚Äî squad rules (2 GK/5 DEF/5 MID/3 FWD, max 3/nation, 100M); `score_players`, `build_fantasy`, `estimate_price`, form/availability filtering, budget-reserve greedy pick.
- `advisor.py` ‚Äî personal advice from `data/my_team.json`. Tolerant name matching (surname + accent-strip via `_make_resolver`/`_squad_identity`). Outputs starting XI, captain, `transfer_options` (per position: weakest out + 2 candidates), `suggest_transfers`, and `differentials` (low-ownership <5%, nailed-on starters, per position ‚Äî counts `config.DIFFERENTIAL_COUNTS` = 3 GK / 5 DEF / 5 MID / 3 FWD ‚Äî from `scraper.fetch_differentials` over the full 48-nation pool; `differentials_for_user` excludes squad by name+surname). The report's fantasy section is now **just** personal advice (pitch) + differentials pitch ‚Äî the generic FIFA-Fantasy/plan blocks were removed.
- `planner.py` ‚Äî fantasy plan; called with `num_matchdays=1` (upcoming matchday only ‚Äî keeps the report short).
- `predictions_log.py` ‚Äî saves the user's predicted scores, settles them against real results, computes hit-rate (outcome + exact) for user **and** model. File: `data/my_predictions.json`.
- `report.py` ‚Äî jinja2 HTML report + Telegram message (`build_telegram_text`) + email fallback. `_within_days` shows all matches in the next `REPORT_UPCOMING_DAYS` (=5) days; `_pitch_rows` renders the lineup as a formation on a CSS pitch; `_differentials_split` shows differentials as 2-per-position on the pitch + 1-per-position on the bench.
- `state.py` ‚Äî cadence / change-detection (decides whether to send).
- `telegram_intake.py` ‚Äî the autonomous bot. `run_bot_once()` (cloud entry via `--bot`), `classify_image` (one Vision call: lineup vs fixtures), `_handle_lineup`, `_handle_fixtures`, `_handle_text` (free Gemini chat about the lineup via `GeminiClient.ask_text`), `_maybe_refresh_model` (gated ~every 5h: pulls **official** results + refreshes the official pool/differentials/fixture-difficulty ‚Äî **Gemini-independent**, so results keep updating even with no quota; Gemini xG enrichment runs only if enabled; then settles the user's predictions). `_handle_fixtures` settles against the latest results before replying so the cumulative **"you vs system" score** in `format_summary_he` is current the moment a predictions screenshot is uploaded; it also reports how many predictions were read (`read`) and lists any bet games **not found in the system** so a missed/mis-read game is visible. **Durability:** if Gemini Vision is unavailable when an image arrives (quota 429), the bot no longer drops the upload ‚Äî `_save_pending_image` stores it under `data/pending_images/` and `_process_pending_images` (run at the start of every `run_bot_once`) classifies + handles it once quota returns, so predictions/lineups are never lost across a quota outage. Also `process_incoming` (legacy photo-only).
- `main.py` ‚Äî pipeline entry: `run()` ‚Üí intake ‚Üí scrape ‚Üí predict ‚Üí fantasy ‚Üí plan ‚Üí advice ‚Üí report ‚Üí telegram.

## Gotchas (important)
- **Gemini free-tier daily quota (429):** image reading and refresh need quota; refresh is gated to ~every 5h. Quota resets daily. Code degrades gracefully (never throws).
- **GitHub schedule cron is unreliable** for frequent intervals ‚Äî that's why the external Apps Script trigger exists.
- **Name matching:** screenshots give short surnames; `db` has full names. `advisor` normalizes surname + strips accents ‚Äî preserve this when touching matching.
- **UTF-8 on Windows PowerShell:** always set `PYTHONUTF8=1`; write files with `encoding="utf-8"`.
- **NEVER commit `.env`** (it holds the secrets). It is git-ignored ‚Äî keep it that way. Do not echo secret values.
- **`data/*.json` is public** (squad visible) but contains **no secrets** ‚Äî tokens/keys are only in `.env` + encrypted GitHub Secrets + the user's Apps Script.
- **PAT expiry (~Sept 2026):** when the Apps Script trigger stops, regenerate the GitHub fine-grained token and update it in the script.
- **Local Windows scheduled tasks are disabled** ‚Äî the cloud is the single source of truth (avoid double-processing Telegram updates).
